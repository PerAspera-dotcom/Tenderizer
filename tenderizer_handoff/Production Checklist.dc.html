Architecture — decide first

Split static frontend from always-on backend (API + engine + DB).
Choose tenancy model: shared DB + tenant_id (recommended start).
Define API base URL as an env var per environment (dev / staging / prod).
Pick the always-on host (see hosting note).
1

Backend — API & engine

Persist run metadata → last_run.json (§5).
Add status column + separate pipeline table (workflow state).
Add config writers for cpv.yaml / keywords.yaml.
Build the ~12 FastAPI endpoints (read · run · pipeline · follow-up).
Connector interface so new portals plug in cleanly (TED/BOAMP pattern).
Keep the 52 engine tests green — pytest -q after each change.
2

Multi-tenancy & auth

User accounts + login (email/pw or SSO).
Tenant isolation — tenant_id on every row / row-level security.
Roles: analyst · manager · admin.
Audit trail — who changed status / notes / deadlines.
Per-tenant config: each company its own CPV set, keywords, portals.
3

Frontend (React + Vite)

Recreate the mockup wired to the live API (don't ship the HTML).
Build order: Dashboard → Feed → Review → Portal → Config.
Env-based API base URL · auth-gated routes.
Leave all 🚧 screens stubbed exactly as the mockup shows.
4

Data & reliability

Migrate SQLite → Postgres before going multi-tenant.
Automated daily backups + a tested restore.
Scheduled scrape (cron) with per-source health logging.
Alert when a source fails N consecutive days.
Secrets & config out of code (.env / secrets manager).
5

Notifications & workflow

Deadline alerts → email / Slack, not just on-screen banners.
Per-tenant new-tender digest.
Pipeline owner assignment + handoff.
6

Compliance

Per-portal ToS review before each goes live (with co-dev co.).
DTVP stays paused until cleared.
GDPR: retention policy for stored notices + user data.
7

Phase 2 readiness

Vault: document store + metadata extraction Scout/Composer reuse.
Composer: requirement-validation gate → locked draft generation.
Keep both preview-only in the app switcher until built.
Critical path: 0 Architecture → 1 Backend → 2 Tenancy → 3 Frontend. Everything else hardens around it.

Technical Decisions & Recommended Stack

The choices that unlock the multi-company goal. Make these before code starts.

p.2
A

API & tenancy

→ One API domain (api.tenderizer.app); tenants resolved by login token — not a URL per customer.
→ Frontend reads the base URL from VITE_API_BASE_URL — one build, per-env config (dev / staging / prod).
→ Shared Postgres + tenant_id on every row (row-level isolation); per-tenant CPV / keywords / portals.
→ Lock CORS to known frontend origins; API on a subdomain, not a path on the marketing site.
B

Recommended stack

Frontend
React + Vite + TypeScript · TanStack Query · React Router
Backend
FastAPI · Postgres · SQLAlchemy + Alembic (migrations)
Auth
Clerk / Auth0 / Supabase Auth — managed; don't roll your own
Scheduled jobs
APScheduler (start) → Celery if it grows · runs run_pipeline
Infra glue
Docker · Sentry (errors) · uptime monitor · secrets manager
C

Hosting — recommended path

Frontend → Vercel
Static deploy, cheap/free, global CDN.
Backend + DB → Railway / Render
Managed always-on container + managed Postgres with automated backups.
Own the domain + database yourself regardless of host — that's the asset.  ·  VPS (Hetzner/DO) = cheaper but you run backups · Big cloud (AWS/Azure) = later, at scale.
D

Expandable scraping — most important for sell-to-many

→ Make the connector a formal interface now: input a portal config → output records in the engine's existing normalized shape.
→ ted.py & boamp.py are your two reference implementations — adding a portal = one new connector file against the contract.
→ Per-tenant portal selection: each company turns on only the sources relevant to them.
Scout = POC proving viability (no tender capture → no automated offers). The full Portal — Scout + Vault + Composer — is the product.
